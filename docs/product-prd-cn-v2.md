更新时间：2026-05-15  
文档状态：基于当前代码实现梳理，并补充下一阶段产品规划  
适用对象：产品、设计、iOS 开发

---

## 1. 产品概述

### 1.1 产品名称

Glug & Poop

### 1.2 产品定位

Glug & Poop 是一款面向个人用户的轻量健康记录 App，围绕“输入”和“输出”两类日常行为做低门槛记录与回看：

1. 输入：喝水、饮品、食物
2. 输出：排便

它当前更接近一个带有趣味表达风格的 `本地单机记录型 MVP`，而不是医疗产品或完整健康管理平台。

### 1.3 产品价值

当前版本的核心价值有三点：

1. 让用户能非常快地完成健康行为记录
2. 让用户能按今天和按日期回看自己的记录
3. 为后续统计、趋势、提醒和洞察能力积累结构化数据

---

## 2. 当前 App 介绍

### 2.1 一句话介绍

这是一个用 SwiftUI 开发的 iOS 记录应用，用户可以快速记录喝水、喝饮料、吃东西和排便情况，并通过 Dashboard 和 Calendar 回顾每天的行为。

### 2.2 当前已实现能力

基于当前代码，App 已实现以下能力：

1. 首页四类记录入口：Water / Drinks / Food / Poop
2. 四类记录的新建流程
3. 已有记录的编辑流程
4. Dashboard 查看“今天”的全部记录
5. Calendar 按月浏览、按天查看记录
6. 记录图片添加与预览
7. 排便记录的结构化录入
8. 本地规则驱动的轻量“AI 吐槽”反馈
9. 基于记录数量的日历活跃度色块

### 2.3 当前未实现或明显不足

1. 没有本地持久化，App 重启后数据会丢失
2. 没有删除记录能力
3. 没有账号、云同步、导出
4. 没有真实统计面板和趋势分析
5. 当前 AI 文案并非模型能力，而是本地规则文案

## 3. 用户与场景

### 3.1 目标用户

1. 有轻量健康自我管理需求的个人用户
2. 想观察饮食、饮水、排便关系的用户
3. 不希望使用复杂表单、偏好卡片化快速记录的用户

### 3.2 核心使用场景

1. 喝水后快速记录本次饮水量
2. 吃饭或喝饮料后补记当次摄入
3. 排便后记录形状、颜色、时长和备注
4. 晚上回看今天是否喝水不足、是否排便异常
5. 在日历中查看某一天的历史行为

---

## 4. 产品目标

### 4.1 短期目标

把当前原型升级成可持续使用的正式 MVP。

短期目标包括：

1. 数据不丢失
2. 记录可完整管理
3. 回看结果有基础价值

### 4.2 中期目标

在记录稳定后，逐步补齐：

1. 基础统计
2. 提醒与目标
3. 更完整的字段结构

### 4.3 长期目标

将产品从“行为打点工具”演进为“个人健康行为洞察工具”。

## 5. 当前信息架构

当前信息架构可概括为：

1. 首页
2. 新建/编辑记录弹层
3. Dashboard 今日记录页
4. Calendar 月历与当日记录页

### 5.1 首页

首页展示四个主入口：

1. Water
2. Drinks
3. Food
4. Poop

底部有两个导航入口：

1. Dashboard
2. Calendar

### 5.2 Dashboard

Dashboard 聚焦“今天”：

1. 展示今日全部记录
2. 按时间倒序排列
3. 点击某条记录进入编辑
4. 底部展示一条反馈文案

### 5.3 Calendar

Calendar 聚焦“按月浏览 + 按天回看”：

1. 月份切换
2. 日期网格
3. 活跃度色块
4. 选中某日后查看当天全部记录
5. 直接从当天视图新增记录

## 6. 功能需求

## 6.1 记录创建

### FR-001 新增喝水记录

用户可以新增一条喝水记录。

字段：

1. 饮水量（0-1500ml）
2. 日期
3. 时间
4. 备注
5. 图片

交互要求：

1. 支持快捷值选择：100 / 300 / 500 / 750 / 1000
2. 支持滑动选择水量
3. 保存后立即在 Dashboard / Calendar 可见

### FR-002 新增饮品记录

用户可以新增一条饮品记录。

字段：

1. 饮品类型
2. 日期
3. 时间
4. 备注
5. 图片

当前预设类型：

1. Coffee
2. Boba
3. Soda
4. Matcha
5. Wine
6. Beer

### FR-003 新增食物记录

用户可以新增一条食物记录。

字段：

1. 食物类型
2. 日期
3. 时间
4. 时长
5. 备注
6. 图片

当前预设类型：

1. Burger
2. Salad
3. Pizza
4. Sushi
5. Tacos
6. Ramen
7. Meat
8. Sweet

### FR-004 新增排便记录

用户可以新增一条排便记录。

字段：

1. 便便形状
2. 便便颜色
3. 日期
4. 时间
5. 时长
6. 备注

说明：

1. 当前排便形状使用 Bristol Stool Scale 图片表达
2. 当前颜色为结构化离散选项
3. 当前排便记录默认不支持图片添加

## 6.2 记录编辑与管理

### FR-005 编辑记录

用户可以编辑任意一条已有记录。

要求：

1. 进入编辑页时自动回填原记录内容
2. 修改后保存，原记录按 ID 更新
3. 如果编辑了时间，记录需重新进入新的日期分组与排序位置

### FR-006 删除记录

用户应可以删除任意一条已有记录。

当前状态：

1. 未实现

需求要求：

1. 删除前二次确认
2. 删除后 Dashboard 与 Calendar 同步刷新
3. 删除后当日统计与活跃度同步更新

## 6.3 查看与回顾

### FR-007 查看今日记录

用户可以在 Dashboard 查看今天的全部记录。

要求：

1. 按时间倒序展示
2. 展示主内容、时间、备注、时长、图片
3. 点击记录进入编辑

### FR-008 按日查看历史记录

用户可以在 Calendar 中查看某一天的全部记录。

要求：

1. 支持切换月份
2. 点击某一天查看当天记录
3. 支持左右滑动查看同月不同日期
4. 支持在当天视图中直接新增记录

### FR-009 日历活跃度标记

系统应根据当天记录条数，在日历单元格上显示活跃度状态。

当前规则：

1. 0 条记录：不高亮
2. 1 条记录：Level 1
3. 2 条记录：Level 2
4. 3 条记录：Level 3
5. 4-5 条记录：Level 4
6. 6 条及以上：Level 5

## 6.4 反馈与轻分析

### FR-010 反馈文案

用户完成记录后，系统可展示一条即时反馈文案。

当前状态：

1. 本地规则生成
2. 不依赖模型服务

### FR-011 Vibe 分值

系统维护一个轻量心情/活跃度分值。

当前规则：

1. 默认值为 70
2. 饮品记录：-2
3. 食物记录：+1
4. 排便记录：+3
5. 喝水记录：不调整

说明：

1. 该值当前只在内存中存在
2. 该值尚未形成独立展示模块

## 7. 数据需求

### 7.1 当前数据模型

当前记录统一为 `LogItem`，核心字段包括：

1. id
2. type
3. detail
4. note
5. duration
6. imageDataList
7. date

### 7.2 数据问题

当前版本存在以下关键问题：

1. 数据只保存在内存
2. 只有一个主文件承载主要逻辑
3. detail 字段承载的结构化程度不统一
4. 图片直接以 Data 挂在记录对象中，长期扩展成本较高

### 7.3 下一阶段数据需求

### FR-012 本地持久化

系统必须支持本地数据持久化。

要求：

1. App 启动时自动加载
2. 新增、编辑、删除后自动保存
3. 记录与图片关联关系明确

### FR-013 数据兼容

系统应兼容旧版排便记录值。

说明：

1. 当前代码已兼容旧 emoji 值映射到新 shapeID
2. 后续持久化迁移时需保留该兼容策略

### FR-014 数据索引

系统应保留按日索引，保证日期查询性能稳定。

---

## 8. 非功能需求

### 8.1 性能

1. 日历切月应保持流畅
2. 当天记录查询应避免全量扫描
3. 图片展示不应造成明显卡顿

### 8.2 可维护性

1. 数据模型、视图模型、页面视图应逐步拆分
2. 各记录类型规则不应长期集中在单一文件
3. 图片处理、持久化、统计逻辑应模块化

### 8.3 可扩展性

1. 支持后续增加统计模块
2. 支持后续增加提醒模块
3. 支持后续增加导出与同步能力

## 9. 版本规划

### V1.1 可用版

目标：从演示原型升级到可日常使用。

范围：

1. 本地持久化
2. 删除记录
3. 数据恢复与兼容
4. 基础稳定性修复

验收标准：

1. 重启 App 后记录仍存在
2. 四类记录都支持新增、编辑、删除
3. Dashboard 与 Calendar 数据一致

### V1.2 复盘版

目标：让数据开始具备回顾价值。

范围：

1. 今日统计
2. 周统计
3. 分类统计
4. 更完整的记录字段

验收标准：

1. 用户可以看到基础统计结果
2. 用户可以更完整记录饮食与排便状态

### V1.3 习惯版

目标：提升持续记录率。

范围：

1. 喝水提醒
2. 晚间补记提醒
3. 简单目标系统

## 11. 结论

Glug & Poop 当前已经具备较完整的 `记录 -> 查看 -> 编辑 -> 回看` 主流程，也有鲜明的产品语气和视觉个性。但从产品成熟度看，它仍处在“高完成度原型”阶段。

下一阶段最重要的不是继续堆新入口，而是先补齐：

1. 持久化
2. 删除
3. 统计
4. 数据结构清晰化

只有这四件事完成后，这个 App 才能从“好看的 demo”真正进入“可以每天使用的产品”阶段。
