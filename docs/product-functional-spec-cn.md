# Glug & Poop 产品功能文档

更新时间：2026-05-15

适用范围：基于当前代码实现整理，不包含 UI 视觉规范，重点描述现有产品功能、数据模型、业务流程和能力边界。

---

## 1. 产品定位

`Glug & Poop` 是一个本地化的个人日常记录应用，核心用途是记录和查看四类健康相关行为：

1. 喝水
2. 饮品摄入
3. 食物摄入
4. 排便记录

应用当前更接近“个人自我记录工具”，而不是完整的健康管理平台。现阶段没有账号体系、没有云同步、没有后端、没有长期分析引擎，所有能力都围绕“快速记录 + 当日查看 + 日历回看 + 轻量反馈”展开。

---

## 2. 当前产品目标

基于现有实现，可以归纳出当前版本的产品目标：

1. 让用户以尽可能低的操作成本完成每日行为记录
2. 支持按天查看和回顾记录
3. 将排便记录从简单文本提升为结构化记录
4. 用轻量反馈提升记录趣味性和连续性

---

## 3. 目标用户

当前版本隐含的目标用户是：

1. 需要记录饮水、饮食、排便情况的个人用户
2. 有基础自我观察需求的用户，例如关注消化、饮水习惯、作息与输入输出关系
3. 更接受轻量、有趣、低门槛记录方式的用户

当前产品不适用于：

1. 多人协作
2. 医疗级诊断
3. 专业健康报告导出
4. 跨设备同步记录

---

## 4. 功能结构总览

当前 App 的功能结构可以分为 5 个模块：

1. 记录创建模块
2. 记录编辑模块
3. 今日汇总模块
4. 日历查看模块
5. 轻量反馈模块

---

## 5. 功能模块说明

## 5.1 记录创建

用户可新建 4 类记录：

1. 喝水记录
2. 饮品记录
3. 食物记录
4. 排便记录

入口位置：

1. 首页四个主入口
2. 日历页下方记录区的四个快捷新增按钮
3. 历史某日详情弹层中的新增入口

创建完成后，记录会直接写入内存中的记录列表，并立即可在 Dashboard / Calendar 中查看。

---

## 5.2 记录编辑

当前支持对已存在的记录进行编辑。

编辑入口：

1. Dashboard 中点击某条记录
2. 日历页下方的记录卡片点击进入编辑
3. 某日详情页中点击记录

可编辑字段：

1. 记录主内容
2. 备注
3. 时间
4. 时长
5. 图片列表

说明：

- 当前没有删除功能
- 编辑本质上是“按 ID 更新旧记录”
- 更新后记录会根据新时间重新排序

---

## 5.3 今日汇总查看

Dashboard 模块用于查看“今天”的记录列表。

当前能力：

1. 拉取当天所有记录
2. 按时间倒序展示
3. 每条记录展示主要内容、时间、备注、时长、图片
4. 支持点击进入编辑
5. 页面底部展示一条轻量的“AI 吐槽/反馈”

说明：

- Dashboard 当前只看“今天”
- 没有筛选、搜索、统计汇总、趋势分析

---

## 5.4 日历查看

Calendar 模块提供按月查看和按天查看能力。

当前能力：

1. 切换月份
2. 查看某月的日期网格
3. 根据当天记录数量生成情绪色块
4. 点击某一天，切换下方记录区内容
5. 左右切换日期内容页
6. 上下滚动查看当天的全部记录
7. 在记录区直接新增四类记录
8. 点击当天记录进入编辑

当前日历页不是简单“打点”，而是“月视图 + 当天记录详情”的组合页。

---

## 5.5 轻量反馈

应用内有两种轻量反馈逻辑：

1. AI 文案反馈
2. Vibe 分值 / 日历情绪等级

### AI 文案反馈

当前不是大模型调用，而是本地规则文案切换。

当新增记录时：

1. 喝水：更新一条吐槽文案
2. 饮品：更新吐槽文案，并降低分值
3. 食物：更新吐槽文案，并提升分值
4. 排便：更新吐槽文案，并提升分值

### Vibe 分值

当前维护一个内存分值 `vibeScore`，默认值为 `70`。

现有规则：

1. 饮品记录：`-2`
2. 食物记录：`+1`
3. 排便记录：`+3`
4. 喝水记录：不调整分值

说明：

- 分值仅存在内存中
- 没有历史趋势
- 没有持久化
- 没有单独的分值页面

### 日历情绪等级

当前按“某日记录条数”粗略计算情绪等级：

1. 0 条：无等级
2. 1 条：Level 1
3. 2 条：Level 2
4. 3 条：Level 3
5. 4-5 条：Level 4
6. 6 条及以上：Level 5

说明：

- 该能力当前是记录活跃度映射，不是真实健康评分

## 6. 四类记录的功能定义

## 6.1 喝水记录

字段：

1. 水量
2. 备注
3. 时间
4. 图片列表

业务规则：

1. 水量范围为 `0-1500ml`
2. 当前步进粒度为 `25ml`
3. 提供快捷值：`100 / 300 / 500 / 750 / 1000`
4. 保存时主内容以字符串形式写入，例如：`500ml`

当前支持：

1. 拍照或从相册添加多张图片
2. 编辑时回填已选图片

---

## 6.2 饮品记录

字段：

1. 饮品类型
2. 备注
3. 时间
4. 图片列表

可选饮品：

1. Coffee
2. Boba
3. Soda
4. Matcha
5. Wine
6. Beer

业务规则：

1. 当前只能单选一种饮品
2. 保存时主内容直接写入饮品名称

---

## 6.3 食物记录

字段：

1. 食物类型
2. 备注
3. 时间
4. 时长
5. 图片列表

可选食物：

1. Burger
2. Salad
3. Pizza
4. Sushi
5. Tacos
6. Ramen
7. Meat
8. Sweet

业务规则：

1. 当前只能单选一种食物
2. 时长范围为 `0-120 min`
3. 保存时主内容直接写入食物名称

---

## 6.4 排便记录

字段：

1. 便便形状
2. 便便颜色
3. 备注
4. 时间
5. 时长

当前结构化能力：

1. 形状支持 8 种
2. 颜色支持 8 种
3. 详情值会被编码成 `shapeID|colorID`

示例：

- `poop_4|coffee`

兼容逻辑：

1. 支持旧版 emoji 记录值向新版 shapeID 自动兼容映射
2. 默认颜色为 `coffee`

当前限制：

1. 排便记录界面关闭了图片上传入口
2. 虽然数据结构上仍保留 `imageDataList` 字段，但该页面当前不提供图片选择

---

## 7. 数据模型

当前核心数据模型为 `LogItem`。

字段定义：

1. `id`：唯一标识
2. `type`：记录类型，枚举值为 water / drink / food / poop
3. `detail`：主记录内容
4. `note`：备注
5. `duration`：时长
6. `imageDataList`：图片数据数组
7. `date`：记录时间

补充说明：

1. `imageData` 是从 `imageDataList.first` 推导出的首图快捷访问
2. 当前排序逻辑基于 `date` 倒序
3. 记录以“天”为单位被建立二级索引，用于日历和当天查询

---

## 8. 数据处理逻辑

## 8.1 新增记录

新增流程：

1. 用户在某入口打开对应记录面板
2. 输入主记录内容、时间、备注等字段
3. 点击提交
4. 生成 `LogItem`
5. 插入总记录列表
6. 同步插入按日索引
7. 触发反馈文案更新

---

## 8.2 编辑记录

编辑流程：

1. 用户在 Dashboard / Calendar 中点击记录
2. 打开对应编辑面板
3. 原记录字段回填
4. 修改并提交
5. 用同一 `id` 构造新记录对象
6. 从原列表和原日期索引移除旧记录
7. 插入新记录并重新排序

---

## 8.3 按日查询

当前系统有按天索引缓存：

1. 每条记录按 `startOfDay(date)` 存入 `logsByDay`
2. Dashboard 直接取今天的记录
3. Calendar 直接取选中日期的记录

价值：

1. 避免每次都全量遍历 `logs`
2. 日历切换和当天展示成本较低

---

## 8.4 月历生成

当前月历数据按月缓存：

1. 输入某个月份
2. 计算该月第一天、星期位置、总天数
3. 生成月视图数组
4. 缓存到 `calendarGridCache`

说明：

- 当前缓存只覆盖月格子结构，不覆盖记录内容

---

## 9. 图片能力

当前图片相关能力：

1. 水、饮品、食物记录支持多图上传
2. 支持从相机或相册获取图片
3. 图片会在输入面板中横向预览
4. 支持移除单张图片
5. Dashboard 和 Calendar 卡片中只展示首图

技术规则：

1. 图片压缩为 JPEG，质量约 `0.6`
2. 图片以二进制 `Data` 形式保存在内存中
3. 使用 `NSCache` 做图片解码缓存

当前限制：

1. 无磁盘持久化
2. 无云端存储
3. 无图片裁剪流程管理
4. 无多图详情浏览

---

## 10. 当前产品边界

当前版本明确“有”的能力：

1. 四类记录的新增
2. 历史记录编辑
3. 今日记录查看
4. 日历按天回看
5. 多图附件能力（部分记录类型）
6. 结构化排便记录
7. 轻量反馈和活跃度映射

当前版本明确“没有”的能力：

1. 数据持久化
2. 本地数据库
3. iCloud / 云同步
4. 登录账号
5. 删除记录
6. 搜索记录
7. 过滤记录
8. 统计图表
9. 导出报告
10. 通知提醒
11. 标签系统
12. 目标系统
13. 健康建议引擎
14. 真正的 AI 分析

## 11. 当前实现的产品风险与缺口

从产品功能角度，当前最明显的缺口有以下几类。

## 11.1 数据可靠性风险

1. 所有数据都在内存中，关闭 App 后记录会丢失
2. `vibeScore` 和 AI 文案也不会被保存

这是当前版本最大的产品问题，因为它直接影响“记录类产品”的基本可用性。

## 11.2 生命周期不完整

1. 有新增
2. 有编辑
3. 没有删除
4. 没有归档
5. 没有恢复

这意味着记录管理闭环还不完整。

## 11.3 健康语义仍偏弱

1. 日历情绪等级只是按记录条数判断
2. `vibeScore` 没有和真实健康逻辑挂钩
3. 排便记录虽然结构化了，但没有进一步分析

因此当前更像“行为打点器”，还不是“健康洞察产品”。

## 11.4 记录粒度受限

1. 饮品和食物目前都是单选，缺少数量、分量、补充字段
2. 喝水只有体积，没有来源或场景
3. 排便没有更多症状字段，如气味、疼痛、频率、紧急程度

## 11.5 缺少长期使用机制

1. 没有提醒
2. 没有连续记录激励
3. 没有周报/月报
4. 没有目标进度

## 12. 推荐的产品功能演进方向

如果基于当前代码继续发展，建议按以下顺序推进。

### 第一阶段：补足可用性基础

1. 接入本地持久化
2. 增加删除记录
3. 补充记录详情页或统一详情能力
4. 增加基础数据迁移策略

### 第二阶段：补足健康记录价值

1. 给喝水加入每日总量统计
2. 给饮品和食物加入数量/分量字段
3. 给排便记录加入更多症状字段
4. 生成按天/周/月统计

### 第三阶段：形成产品闭环

1. 提醒系统
2. 连续打卡和目标系统
3. 周报/月报
4. 更合理的健康评分体系

### 第四阶段：增强数据能力

1. 本地数据库化
2. 云同步
3. 导出与分享
4. 真正的 AI 分析能力

---

## 13. 当前版本结论

从功能上看，当前 App 已经完成了一个可演示的最小闭环：

1. 可新增记录
2. 可编辑记录
3. 可按日查看
4. 可在日历中浏览历史
5. 可做基础的结构化排便记录

但它仍处于“高保真原型 / 本地单机 MVP”阶段，而不是可长期使用的正式产品。决定产品是否能进入下一阶段的关键，不是继续堆 UI，而是优先补齐以下三项：

1. 数据持久化
2. 记录生命周期完整性
3. 更可信的健康语义和统计能力
